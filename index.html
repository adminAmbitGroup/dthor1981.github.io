Structural Design

Servicenow provides tools to create a modular pattern with apps. Therefore, we have incorporated modularity in our architecture to keep the units of code both cleanly seperated and organized. However, since some files can be accessed globally, you may need to adhere to the naming conventions in the “Coding Practices” below to ensure the pattern is uniform.

Coding practices

Servicenow uses both global as well as local application scopes to share and separate concerns from one app to the other. Because of this, you should use the naming conventions described below.

File Naming

When adding files(including content blocks) to apps, you are expected to follow these guidelines

  • Global
    • Keyword Prefix: Prmmt
      • ***Global Script UI do not need Prefix***
    • PascalCase – Start with uppercase
    • Use underscores and no spaces in between words
    • Used for all global blocks, script names, macro,etc.
    • Examples:Prmmt_Tbl_Script_Inc, Prmmt_Static_Block, Prmmt_Dyn_Block
  • Local
    • Keyword Prefix:
      • Grants Bi: Gbi
      • QASP: Qasp
      • CCMS: Ccms
      • Tagr: Tagr
      • ***Local Script UI will add prefix app scope name before file.***
    • PascalCase – Start with uppercase
    • Use underscores and no spaces in between words
    • Used for all local blocks, script names, macro,etc.
    • Examples:Gbi_Tbl_Script_Inc, Qasp_Static_Block, Ccms_Dyn_Block

API Script Syntax and Semantics:

When working within scripts in Servicenow, please use common and recommended programming syntax. Be mindful when using spaces, hyphens and underscores. Please remember to…

//////////////////////////////////////////////
//COMMENT YOUR CODE//
/////////////////////////////////////////////

When working with semantics in Servicenow, feel free to use the prefixes below to reference the local app.

  • Keyword Prefix:
    • Grants Bi: gbi
    • QASP: qasp
    • CCMS: ccms
    • Tagr: tagr
    • ***Please note prefix begins with lowercase instead of uppercase like the filenames.***

    • Although this is not necessary in every use case, it is appropriate when distinguishing between applications that may use similar functions such as “gbiCreateBarChart()” and “qaspCreateBarChart()”. If an application requires the use of scopes from two or more applications then a custom prefix should be created. For instance, when referencing a table from the Qasp app as well as a Grants Bi app.

    ServiceNow Administration

    Sets

    For every application, each developer needs their own update set. It is critical every change you make for an application is done in the update set created for that purpose. If not, the application will not push to production correctly and it will require a rebuild or major fix.

    For example: We are creating a new TAGR application and Navy and I are working it. There will be 3 update sets for TAGR.

    1. TAGR Default
    2. TAGR Winston
    3. TAGR Navy

    Every change I make working on TAGR needs to be done while I am in the “TAGR Winston” update set. This requires constant vigilance and is made easier by adding the update set picker to the top navigation bar (settings >developer>show update set picker). Only my TAGR works goes in that update set. If I need to work on GBI, I have to switch update sets before I do any work.
